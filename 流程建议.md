# 流程建议

本文档记录当前项目的执行路线。最初阶段的环境选择、是否租算力、是否微调大模型等问题已经有结论，这里不再展开长篇分析，只保留当前关键配置和后续操作策略。

## 1. 当前结论

当前应按照“先本地稳定验证，再谨慎提交官网”的路线推进。

也就是说：

1. 不把每次小改动都提交到官网。
2. 每次实验先在本地 5 折交叉验证。
3. 只有当本地 AUC 明显、稳定超过 baseline，才考虑占用官网提交次数。
4. 每次代码或特征修改都记录到 `实验记录.md`。
5. 每个成熟阶段再做 Git 提交，方便回退和对比。

与最初流程相比，当前已经完成了第一版 baseline，所以后续重点不再是“如何跑通”，而是“如何稳定提升”。

## 2. 当前比赛任务

任务类型：

```text
金融风控表格二分类
```

目标：

```text
预测贷款用户是否违约
```

标签列：

```text
isDefault
```

提交内容：

```text
testA.csv 中每个 id 对应的 isDefault=1 概率
```

评价指标：

```text
AUC，越高越好
```

AUC 衡量的是模型对风险高低的排序能力。模型给真正违约用户的预测概率越倾向于高于未违约用户，AUC 越高。

## 6. 官网提交策略

官网提交次数有限，因此不采用“每改一次就提交”的方式。

建议把当前 baseline 作为对照：

```text
baseline AUC = 0.737236
```

后续版本至少满足下面条件之一，再考虑提交官网：

- 本地 5 折 OOF AUC 达到 `0.739+`
- 本地 5 折 OOF AUC 虽提升不大，但特征逻辑清晰且每折都稳定提升
- 得到一个和 LightGBM 差异较大的模型，例如 CatBoost，并且本地表现接近或超过 baseline
- 融合模型比所有单模型更稳定

更理想的提交门槛：

```text
本地 OOF AUC 达到 0.741+ 后再提交一次
```

计划保留 5 次官网提交的大致用途：

1. 成熟 LightGBM 特征增强版
2. 调参后的 LightGBM 强化版
3. CatBoost 或 XGBoost 单模型
4. LightGBM + CatBoost 融合
5. 最终融合版本

实际提交顺序以后根据本地实验结果调整。

## 7. 后续执行流程

后续每一轮实验按以下流程执行：

1. 明确本轮目标，例如“新增统计特征”或“调整 LightGBM 参数”。
2. 修改代码前，在 `实验记录.md` 追加实验条目草稿。
3. 修改脚本。
4. 先跑小样本冒烟测试，确认代码不报错。
5. 跑完整 5 折训练。
6. 检查提交文件格式和预测分布。
7. 把 AUC、每折结果、耗时、输出文件写回 `实验记录.md`。
8. 和 baseline 对比，决定是否保留。
9. 如果实验值得保留，进行 Git 提交。
10. 如果达到官网提交门槛，再考虑提交官网。

## 8. 第二阶段优先方向

下一步建议先做本地特征分析和 LightGBM 特征增强，不急着换模型。

优先级如下：

1. 查看当前特征重要性。
2. 基于重要特征做交叉、比例、统计特征。
3. 增强日期相关特征。
4. 增强高基数类别的频次、目标编码或分组统计特征。
5. 做 LightGBM 参数小范围调整。
6. 再尝试 CatBoost。
7. 最后考虑模型融合。

第一轮优化建议从下面几类特征开始：

- `grade`、`subGrade` 的等级序号和组合统计
- `issueDate` 的年份、月份、季度、时间趋势
- `earliesCreditLine` 与 `issueDate` 的信用历史长度
- `loanAmnt`、`annualIncome`、`installment`、`dti` 的还款压力特征
- `employmentTitle`、`title`、`postCode` 的频次和分组统计
- 匿名特征 `n0` 到 `n14` 的缺失数量、总和、均值等聚合特征

## 9. 当前不做的事情

暂时不做：

- 大模型微调
- 深度学习模型
- GPU 训练配置
- 自动机器学习
- 大规模无方向调参
- 频繁官网提交

这些事情不是不能做，而是不适合作为当前阶段的优先项。当前最有价值的是稳定验证、可靠记录和逐步增强特征。
